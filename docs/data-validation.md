# 데이터소스 검증 보고서 — food-budget-app (월 식비 예산 기반 밀플래닝)

> 작성 대상: 팀 6인 · 검증 기준일: 2026-07-09 · 범위: design.md §3(데이터소스) 6종 + 연동체인
> 본 보고서는 **팀 공유용 검증 결과**이며, `docs/design.md`(설계 정본)를 수정하지 않는다. 채택/재정의 의사결정은 팀 회의에서 별도로 확정할 것.
>
> **검증 방식:** 17개 병렬 서브에이전트(소스별 웹조사 → 적대적 반증 → 연동체인 폐합검증 → 종합 → 과대표기 심사). 공식 API 문서·robots.txt·ToS·라이선스 확인 + 일부 직접 fetch 재현. **대량 크롤링은 수행하지 않음**(비상업·교육 전제) → 런타임 수율/품질은 🧪 실측필요로 분리.
> **⚠️ 헤드라인:** "전부 검증 완료"가 **아니다.** 6+1 소스 중 **✅ 1(식약처, 접근 한정) · ⚠️ 2(YouTube·영수증) · 🔴 4(쿠팡·만개·냉부·지마켓)**, **연동체인 미폐합(🔴)**, 그리고 **표준 품목코드 정본(master)이 설계에 부재 = 최우선 선행 블로커.** 정직한 결론은 **"데이터 계층에 치명 블로커 다수 발견"**. → §6 + §7(심사노트).

## 0. 이 문서의 상태 정의 (정직하게 먼저)

**이 검증은 "설계·피저빌리티·스키마 검증"이지 "런타임 실측"이 아니다.**

- 확인한 것: 공식 API/문서/robots.txt/ToS/라이선스의 **존재와 스펙**, 소스가 제공하는 **필드와 서비스 조인컬럼의 정합성**, 접근을 막는 **기술·법적 장벽의 실재 여부**(일부는 직접 fetch로 재현).
- 확인하지 **못한** 것: 실제 키를 발급받아 대량 수집했을 때의 **수율·정확도·품질**. 크롤러/헤드리스 우회의 **실제 성공률**. NER/OCR/Gemini 추출의 **실측 성능**.
- 따라서 "🔴블로커" 판정은 "설계된 형태로는 성립 불가"라는 뜻이지 "물리적으로 100% 불가능"의 증명이 아니다. 반대로 "✅검증완료"도 "접근 가능"까지의 보증이지 "품질 확보"의 보증이 아니다.

### 판정 4단계 범례

| 기호 | 판정 | 의미 |
|---|---|---|
| ✅ | **검증완료(피저빌리티)** | 공식·합법 경로로 오늘 접근 가능. 필드·조인 정합. 잔여 리스크는 품질 실측 수준. |
| ⚠️ | **조건부** | 접근은 되나 특정 조건(예외 승인, 설계 재정의, HITL) 충족 시에만 계획대로 사용 가능. |
| 🔴 | **블로커** | 계획된 형태로는 기술적/법적/구조적으로 성립 불가. 소스 재정의 또는 대체 없이는 하류 봉쇄. |
| 🧪 | **실측필요** | 판정을 확정하려면 런타임 실행/키 발급/PoC가 반드시 선행되어야 하는 항목. |

> 판정은 **1차 조사 + 적대적 검증(verifier) 재확인**을 거친 **조정 판정**을 싣는다. 1차와 조정이 다른 경우 본문에 명시한다.

## 2. 소스별 상세

### 2.1 🔴 쿠팡 (신선+가공) — P0, 판정 상향 (1차 조건부 → 블로커)

**무엇이 나오는가.** 접근 가능했다면 파트너스 API가 주는 필드는 `productId / productName / productPrice / productImage / productUrl / rank / isRocket / isFreeShipping`으로 **매우 얇다**. 정가·할인율·카테고리(상품별)·재고·리뷰수·**용량(g/ml/개)**·가격이력은 어느 API도 주지 않고 오직 크롤 HTML에만 존재.

**어떤 컬럼이 서비스와 연동되는가.**
- 제공: 상품ID(ClickHouse 시계열 PK/ES 문서ID 후보), 상품명, 판매가(현재가 1점), 이미지, 수집시각(자체 부여).
- **부분/없음**: 용량·규격은 **전 경로 비구조화**(상품명 텍스트 파싱만) → 100g당 단가 비교·표준 품목코드 매칭의 최대 난관. 가격 **이력**은 API 미제공 → 크롤 스냅샷을 ClickHouse에 **수개월 자체 축적**해야만 P0 최저가 이상탐지·P1 할인주기 예측 학습셋이 생성됨.
- **조인축 붕괴**: 쿠팡 단일 채널이라 "다매장 최저가 비교"가 성립 안 함(로켓/판매자/시점 비교로 한정). design.md의 "가격비교"는 사실상 쿠팡 내부 비교로 축소.

**무엇이 막히는가 (확정 블로커).**
- robots.txt: `User-agent: * → Disallow: /` — 명시 검색엔진(Googlebot/Bingbot/Naver Yeti/Daum)만 화이트리스트, 계획 크롤러 포함 그 외 전 봇 금지. (robots.txt 자체가 Akamai 403이라 원문은 프록시로만 확인) — https://www.coupang.com/robots.txt
- **Akamai Bot Manager 5계층**(TLS핑거프린트→70KB 센서JS/`_abck` 쿠키→브라우저핑거→행동분석→IP평판). requests/curl/Scrapy 차단은 물론 **2024~2026 현재 Selenium/Playwright/Puppeteer 헤드리스도 거의 전면 차단**. 우회엔 주거용/로테이팅 IP 필요 → **학생예산·AWS Spot(데이터센터 IP 조기차단)·비상업** 하드제약과 정면 충돌. — https://blog.hashscraper.com/posts/coupang-crawling-2026-complete-guide-everything-about-akamai-bypass
- 파트너스 API: **키 발급 게이트 = 최종승인(누적 판매 15만원 이상, 자기추천 구매 금지)** → 트래픽 없는 비상업·비공개 앱은 실판매 도달 불가 = 발급 사실상 불가. 발급돼도 **시간당 10회×건당 10개 = ~100상품/시간**(수만 SKU 일2회 갱신을 수학적으로 불가), 한도 초과 3회 시 계정 API 영구 제한, minPrice/maxPrice 가격필터 무시. — https://mg.jnomy.com/coupang-partners-verify , https://codedosa.com/12603
- 셀러(WING) Open API는 vendorId 기반 **자기 등록상품 한정** — 임의 마켓 상품 조회 경로 아님, 대체 불가. — https://developers.coupangcorp.com/

**판정 근거.** 합법 API 경로 전폐쇄 + 유일 볼륨 경로(크롤)가 robots 위반·Akamai 기술차단·우회 인프라가 프로젝트 하드제약과 충돌. **1차 "조건부"는 과대 낙관 → "블로커"가 정직한 판정.** (ToS의 데이터 저장/캐싱 금지 조항은 원문 미확인·저확신이나 블로커 핵심 근거 아님.)

### 2.2 🔴 만개의레시피 — P0, 판정 상향 (AI학습 용도 기준, 1차 조건부 → 블로커)

**무엇이 나오는가.** 요리명, **재료(재료명+분량, 대체로 분리되나 일부 한줄 자유텍스트)**, 조리순서(스텝+이미지), 인분, 조리시간, 난이도(아무나~신의경지), 카테고리 4종, 등록자/조회수/추천수/해시태그. **영양정보 없음**(식약처 조인으로 확보), **표준 품목코드 없음**(CRF NER로 생성 — 설계 전제).

**어떤 컬럼이 서비스와 연동되는가.** 인분/시간/난이도/스텝/이미지는 ES 레시피 인덱스 구성에 충분(제공). 재료명·분량은 **부분** — 자유텍스트 한국어, 비정규화('다진 마늘','청양고추 2개','약 1200g','적당량')로 CRF 정규화 대상. 쿠팡/영양 매핑은 공유키가 아니라 ES nori 유사도로 연결(설계와 일치).

**무엇이 막히는가 (확정 블로커).**
- **권리충돌(핵심)**: robots.txt가 `Content-Signal: search=yes, ai-train=no, use=reference`를 **EU 저작권지침 2019/790 제4조 "권리의 명시적 유보(EXPRESS RESERVATIONS OF RIGHTS)"**로 명문화하고 ClaudeBot·GPTBot·CCBot·Google-Extended·meta-externalagent·Bytespider 등 AI크롤러를 전부 `Disallow:/`. 프로젝트 주 용도인 **CRF NER 학습 입력을 정면으로 겨냥한 기계판독 TDM 옵트아웃**. — https://www.10000recipe.com/robots.txt
- **기술차단(실측 재확인)**: `/recipe/*` 상세와 `/ranking/*` 라이브 페이지 모두 **HTTP 403** 재현. 헤드리스+UA위장 우회는 명시적 옵트아웃을 뚫는 anti-circumvention/무단접근 소지 가중.
- **합법 대체(KADX)도 불완전**: aT 운영 KADX 무료판은 **CC BY-NC-ND-SA**(변경금지=학습·정규화·파생배포 제약, ND+SA 동시표기는 논리모순 비정형 라이선스) + 로그인 게이팅 + **부분 커버리지(12만 vs 전체 19~20만)**. 전체판은 유료·판매자 협의. — https://kadx.co.kr/opmk/frn/productDetail/PRD_0000000000377/1
- CLAUDE.md 스스로 "AI 학습 목적 TDM은 별도 검토"로 법률검토 미완.

**판정 근거.** **순수 레시피 인덱스/검색표시 용도**(search=yes 허용 + KADX 존재)로는 조건부 여지가 있으나, **CRF NER 학습(P0 moat의 주 입력) 용도로는 명시적 ai-train=no 옵트아웃 + 403 + ND 대체채널로 블로커.** 재료 구조(구조화 vs 자유텍스트)는 403으로 라이브 미검증 → NER 약라벨 확보량은 실측 필요.

### 2.3 🔴 냉장고를부탁해 — P0

**결론 먼저: 계획서 §3.1의 "냉장고를부탁해 = 독립 사전 크롤링 정형 레시피 DB(방영 스케줄 연동, 주1회, P0)"에 해당하는 공식·비공식 엔드포인트가 실존하지 않는다.**

- 프로그램 실체는 확인됨(JTBC 예능, 시즌1 2014~2019 종영, 2024-12-15 "since 2014" 부활 방영 중). 그러나 **방송사는 재료·분량을 구조화 공개한 적이 없음** — tv.jtbc.co.kr은 봇차단(403)·영상 메타만. — https://ko.wikipedia.org/wiki/냉장고를_부탁해

### 2.5 ⚠️ 유저 영수증 OCR — P0 (용도별 분리 판정 필수)

**무엇이 나오는가.** 외부 API/크롤이 아닌 **유저 본인 데이터** 파이프라인(인증/robots/ToS 무관, 비상업·비공개와 정합).
- **Tesseract 경로(primary, 무료·로컬 CPU·Apache-2.0)**: 라인별 텍스트 + 단어별 bbox + confidence **만** 제공. 구조화 IE 없음 → 품목-수량-단가-금액 재구성 파싱을 좌표 기반으로 자체 구현해야 함.
- CLOVA 대안(유료): `storeInfo`(name/bizNum), `paymentInfo`(date/cardInfo), `items[].name/code/count/price.price/unitPrice`, `totalPrice/subTotal`을 confidenceScore+boundingPolys+maskingPolys와 함께 구조화 제공. — https://api.ncloud-docs.com/docs/en/ai-application-service-ocr-ocrdocumentocr-receipt

**어떤 컬럼이 서비스와 연동되는가.**
- **제공(무료 툴로 성립)**: 구매일자, 합계, 매장명 → **캘린더 식비** 유즈케이스는 무료 Tesseract로 동작.
- **표준 품목코드 = 없음.** CLOVA `items[].code`도 표준코드 아님(문서상 의미 미명시=추론) + **실물 영수증엔 코드 미인쇄가 잦아 빈번히 공백** → 코드 기반 조인이 상당수 영수증에서 불가.
- 축약명·수량·단가 = 부분(OCR 정확도 종속).

**무엇이 막히는가.**
- **Tesseract 한국어 감열지 부적합**: KORIE(한국 소매 영수증 748매) 기준 **Tesseract CER 25.43% / WER 35.26% — 오픈툴킷 중 최악**(PaddleOCR 15.84%/26.73%). 스캔본에서도 단어 3개 중 1개꼴 오류, 폰카·구겨진 감열지는 더 악화. — https://www.mdpi.com/2227-7390/14/1/187
- **표준 품목코드 수렴 실패**: OCR 오류 + 축약 POS명↔레시피 NER **도메인 시프트** + code 공백이 겹쳐 **냉장고 재고→표준코드 조인은 HITL(사람 확인 UI) 없이는 de-facto 블로커.** (상용 시스템이 OCR 텍스트 대신 바코드로 매칭하는 것이 난이도 방증 — https://patents.google.com/patent/KR101139801B1/ko)
- **품질 위해 CLOVA로 가면 유료 API 절대제약 충돌 확정**: 무료 5,000요청은 General OCR 한정(영수증 Document 모델 미적용), 도메인은 호출 없이도 상시 유지비 발생 → 상시 유료, **Gemini식 별도 예외 승인 필요.**
- **신규(PIPA)**: 영수증은 카드번호 등 민감정보 포함, 서버가 원본 수신·저장 → 마스킹/보관/파기 책임 발생. 무료 Tesseract 경로엔 maskingPolys가 없어 **개인정보 마스킹 자체 구현 필요**(비상업 전제라도 법적 의무 면제 안 됨).

**판정.** **캘린더 식비**(구매일자+합계+매장명)는 무료 툴로 성립 → 조건부. **냉장고재고→표준코드 조인**은 HITL 없이는 de-facto 블로커. **용도별 분리 계획 필수.** 자체 파싱/정규화 성공률은 PoC 미실측(🧪).

## 3. 연동 체인 검증 — **결과: 끊김(체인 미폐합)**

서비스 핵심 흐름은 `레시피 재료명 → CRF NER → 표준 품목코드 → ES nori → 상품(쿠팡/지마켓) 매핑 → 가격비교/장바구니/예산추적/최저가알림 + 영양(식약처)/재고(영수증) 조인`이다. 이 체인은 **닫히지 않는다.**

### 3.1 [치명·근본] 표준 품목코드 공간의 정본(master)이 **어디에도 없다** = 히든 의존성/최우선 블로커

design.md는 '표준 품목코드'를 **CRF NER의 출력으로만** 취급(§4.1 "대파 반 단"→코드, §4.3 "이 품목코드가 연결 키", §6.2 NER결과→ES nori). 그러나 **CRF는 재료명 스팬/라벨을 뽑을 뿐 코드 레지스트리를 생성하지 못한다.** 즉 CRF가 "정규화해 넣을" 코드공간이 **어느 소스에도 존재하지 않는다.** video-recipe-ai.md §S2의 '자체 재료 사전'이 유일한 근접 언급이나 validator 역할로만 등장하고 시드/코드체계/유지주체 미정의.

- **정본이 되어야 할 유일한 현실 후보** = 식약처 식품코드/국가표준식품성분표(data.go.kr 15100064·15100065). 6개 소스 중 유일한 검증완료·무료·이용제한없음·계층분류+대표식품명·상시접근.
- 따라서 master는 **"식약처 식품코드로 앵커링한 자체 품목 마스터(canonical 품목 사전 + 동의어/약어/용량 정규화 규칙)"**를 오프라인 1회 구축·유지하는 형태여야 함.
- **이 '자체 품목 마스터' 산출물이 설계 정본에 아예 부재한 것 자체가 체인을 닫기 위한 선행 블로커.** 이게 없으면 체인이 논리적으로 **시작조차 못 한다.**

### 3.2 명칭공간(name-space) 4종 불일치 — 단일 CRF/nori로 수렴 불가

| 명칭공간 | 프로파일 | 문제 |
|---|---|---|
| 영수증 축약명 | 매장별 POS 약어(브랜드+규격 공백없이 절단, 예 '삼겹500') + OCR 오류(CER~25%) | code 빈번 공백, 레시피 CRF 도메인 시프트 → HITL 없이 수렴 불가 |
| 레시피 재료명 | 자유텍스트 구어체('대파 반 단','적당량') + 비표준 단위 | CRF 정규화 본래 대상이나 **주 학습입력(만개)이 ai-train=no+403으로 봉쇄** = 학습셋 공급원 차단 |
| 쿠팡/지마켓 상품명 | 브랜드+수식어+**용량(g/ml/개/팩)이 텍스트에 임베드** | 핵심 매칭키 용량이 비구조화 + **두 소스 다 접근 블로커라 매칭 대상 텍스트조차 미확보** |
| 식약처 식품명 | 계층 쉼표식 + 대표식품명 병존 | 접근은 깨끗하나 표기 구조가 위 3개와 시프트 → 별도 정규화/동의어 사전 필수 |

**결론**: 4개 명칭공간은 어휘·구조·오류 프로파일이 서로 달라 **단일 CRF 하나 또는 nori 토큰화만으로 하나의 코드공간에 수렴 불가.** 명칭공간별 정규화기 + 공유 품목 마스터 + 동의어/약어/용량 사전이 **모두** 필요.

### 3.3 조인키 갭 정리

- 쿠팡 `productId`·지마켓 `goodsNo`·영수증 `items[].code`·식약처 `식품코드` = **전부 리테일러/POS/기관 내부 식별자, 어떤 것도 표준 품목코드가 아님** → 상호 코드 직결 불가, 전부 이름 기반 매핑만 잔존.
- **용량/규격**이 쿠팡·지마켓·영수증·일부 식품명 전반에서 비구조화 → **100g당 단가 비교(가격비교 핵심 유즈케이스)**의 최대 난관, 별도 용량 파서 필요.
- **매장(멀티리테일러) 조인축 붕괴**: 쿠팡 단일 채널이라 '다매장 최저가 비교' 미성립, 지마켓 결합으로만 성립하나 **지마켓이 블로커** → 다매장 가격비교 축 자체가 데이터로 성립 불가.

### 3.4 ES nori의 한계

ES nori는 **레시피명↔식품명의 느슨한 텍스트 매칭**에 부분 기여할 뿐, **영수증 절단 약어('삼겹500')·상품명 용량 파싱은 흡수하지 못한다.** 별도 매핑 사전/용량 파서 없이는 nori 단독으로 코드공간 수렴 불가. 즉 "nori 유사도로 다 연결한다"는 설계 가정은 **명칭공간별 정규화기가 선행될 때만** 유효.

## 6. 종합 판정

**연동체인 판정: 🔴 끊김(체인 미폐합).**

서비스 본체("월 식비 예산 관리")가 의존하는 **상품 가격 축(쿠팡 P0 + 지마켓 P1)이 둘 다 블로커**이고, moat인 **CRF NER의 주 학습 소스(만개 P0 + 냉부 P0)도 둘 다 블로커**다. 유일하게 상대적으로 성립하는 레그는 하류 '가격/상품' 반쪽이 아니라 **'영양(식약처 ✅) + 캘린더/영수증 보조 레그(⚠️)'뿐**이며, YouTube는 조건부다. 게다가 체인의 중심 조인노드인 **표준 품목코드 정본이 설계에 아예 부재**하여 체인이 논리적으로 시작조차 못 한다.

### 우선 처리 블로커 Top 3

1. **[근본] 표준 품목코드 master(코드공간 정본) 미정의.**
   → **식약처 식품코드로 앵커링한 "자체 품목 마스터"(canonical 품목 사전 + 동의어/약어/용량 정규화 규칙)를 선행 산출물로 설계 정본에 추가·구축.** 이게 없으면 나머지 모든 조인이 공중부양한다. **가장 먼저, 그리고 팀 의사결정으로.**

2. **[치명] 상품 가격 축 전멸 — 쿠팡·지마켓 둘 다 블로커.**
   → 하류 '코드→ES→상품매핑→가격비교/장바구니/예산추적/최저가알림/할인주기예측'이 데이터 공급원 상실. **가격 데이터 소스 전략 재수립 필요**(합법 공개 가격 API 탐색, 유저 크라우드소싱 가격 입력, 또는 서비스 스코프에서 '실시간 크롤 최저가' 축소 등). anti-bot 우회는 학생예산·비상업 하드제약과 충돌하므로 우회를 전제한 계획은 금지.

3. **[치명] NER 학습 상류 봉쇄 — 만개(ai-train=no)·냉부(정형 소스 부재).**
   → moat인 CRF NER의 주 학습 데이터가 법적·기술적으로 막힘. **합법 학습 코퍼스 대안 확보**(KADX 유료 전체판 라이선스 협의, 공공 레시피 오픈데이터 재검토, 또는 유저 제출 YouTube/영수증 기반 자체 약라벨 부트스트랩)와 **CLAUDE.md가 미완으로 둔 "AI 학습 목적 TDM 법률검토" 완료**가 선행되어야 한다.

**차선 처리(⚠️ 조건부 2종):**
- **YouTube+Gemini**: 자막 폴백 → "설명란+Gemini 오디오 전사"로 설계 정정 + Gemini 유료 예외 승인 대비 + 추출 품질 PoC(🧪).
- **영수증 OCR**: **용도별 분리** — 캘린더 식비는 무료 Tesseract로 진행, 냉장고재고→표준코드 조인은 HITL 확인 UI를 필수 전제로. CLOVA는 유료 API 예외 승인 시에만. PIPA 마스킹 자체 구현 필요.

**건강한 레그(✅ 1종):** 식약처 영양DB만 즉시 채택 가능. 다만 이름기반 조인 + 매핑테이블 구축 + 원물 커버리지 실측(🧪)이 조건. 이 소스가 곧 **블로커 1번(품목 마스터 정본)의 앵커**이므로, 실무 착수 1순위로 삼을 것.

## 7. 심사 노트 — 자기검증 정정 + 추가 미검증 항목

> 본 보고서 자체를 적대적으로 심사(overclaim-critic)한 결과. 판정 방향은 뒤집히지 않았고(견고), 아래는 **신뢰도 라벨 정정 + 본문이 놓친 데이터 공백 발굴**이다.

### 7.1 본문 과대표기 정정

- **식약처 ✅ → "✅ 접근·필드·라이선스 / 🧪 조인 매칭률 미검증"으로 분리.** §1 매트릭스·§6의 "유일한 건강한 레그/앵커 확정"은 *접근성* 보증일 뿐, 실사용(구어 재료명 ↔ 계층 쉼표식 표준명 매칭)은 **라이브 매칭 PoC 미실측**. 앵커 채택은 "대파·무 등 대표 재료 라이브 매칭 PoC 선행" 조건.
- **식약처 "7대 영양소 초과충족"은 표준데이터 파일(15100064/65) 경로 한정.** Open API 15127578 응답에는 나트륨/당류/콜레스테롤이 실호출로 확인되지 않음(정적 파일 적재 경로라 서비스 영향은 없음).
- **식약처 호출한도 "10,000/일"은 상향 신청 최대치**, 기본 제공은 ~1,000/일 규모(미확정). 정적 DB라 서비스 영향 없음.
- **만개 "KADX 개방 12만건 / 31컬럼"은 로그인·JS 게이팅으로 미검증**(§5 실측 #3) — 확정치로 인용 금지.

### 7.2 추가 미검증·누락 데이터 공백 (본문이 놓친 축 — 별도 검증 티켓 필요)

본 검증은 design.md §3의 6소스에 집중했으나, 다음 서비스 축은 **데이터 공급원이 아예 평가되지 않았거나 부재**하다:

1. **🔴(신규) 신선도/유통기한 데이터 소스 부재.** P0 기능#2(유통기한 알림) + P1 신선도예측(XGBoost)에 필요한 **유통기한·신선도 라벨을 6소스 중 어느 것도 제공하지 않음.** 이 P0 축의 학습·입력 공급원이 설계에 없음.
2. **🔴(신규) Pantry 안전알림(P0 기능#2) 공급원 미확정.** 식품회수/판매중단(공공데이터 15074318, design §3.2 보류)이 사실상 유일 공급원인데 보류·미평가 → "안전알림 데이터원 미확정" 리스크.
3. **엥겔지수(기능#9 = 식비/소득)의 소득(income) 축 데이터 부재.** OCR은 식비만 채우고 소득 노드는 공란 → 엥겔지수 산출 입력 부족.
4. **레시피 랭킹(P1, LightGBM) 콜드스타트 학습 데이터**(클릭/상호작용 로그) 공급원 미검토.
5. **이커머스 캡쳐 OCR ↔ 차단된 쿠팡 상품명 브릿지 미탐색.** 유저가 올린 이커머스(쿠팡 등) 주문 캡쳐 OCR은 "사용자 구매분 한정" 상품명을 제공 → 차단된 쿠팡 가격축의 **부분 대체(크라우드소싱 가격)** 가능성. §2.1/§3의 "쿠팡 상품 텍스트조차 미확보" 결론을 부분 보완할 잔여 경로로 검토 가치.

### 7.3 근거 약한 인용 (확정치로 신뢰 금지)

## 8. 추가 검증 (후속, 2026-07-09) — 마켓컬리를 쿠팡 가격축 대안으로

> §2.1(쿠팡 🔴)·§6 블로커#2(상품 가격 축 전멸)에 대한 후속. 쿠팡·지마켓이 막힌 뒤 대체 가격 소스로 **마켓컬리(kurly)**를 직접 정찰·필드 실측. 방식: robots 확인 + `requests` 단건 정찰(대량 크롤 없음) + 커뮤니티 후기 조사.

### 8.1 판정: ⚠️ 조건부-사용가능 (회색 — 우회는 아니나 소유자 봇-금지 지시 무시)

**쿠팡과 결정적 차이: 우회할 기술적 anti-bot이 없다.** www·api 모두 CloudFront 서빙, Akamai/Cloudflare 인터랙티브 챌린지 없음(200 정상). 남은 장벽은 **권고성 robots + ToS**뿐 → 기술적 우회가 불필요한 회색지대(쿠팡=기술 우회 + 형사 판례권의 블랙과 급이 다름).

- `www.kurly.com` 상품 페이지: robots **허용** + sitemap 제공. 단 SSR이 `product:null`(FALLBACK) → **가격 데이터가 서버 HTML엔 없음**.
- `api.kurly.com`: **robots `Disallow: /`**(Googlebot만). 실데이터는 여기 JSON → **데이터 접근 = 이 api robots 지시를 무시**하는 것(감수 트레이드오프).

### 8.2 연동 컬럼 — 실데이터로 확인 (쿠팡 3대 난관 해결)

실호출 상품 `5054675 = 깐녹두 500g` 기준:

| 컬리 필드(실값) | 서비스 연동 | 쿠팡 대비 |
|---|---|---|
| `no`=5054675 | ClickHouse PK / ES ID / diff 키 | 동급 |
| `name`="깐녹두 500g" | CRF NER → 표준 품목코드 | 동급 |
| **`volume`="500g"** | 구조화 용량 → 품목코드 정규화 | 🟢 쿠팡 최대 난관(상품명 파싱) 해결 |
| **`unit_price_text`="100g 당 2,980원"** | 100g당 단가(가격비교 핵심) | 🟢 쿠팡 불가 → 컬리가 계산 제공 |
| `base_price`=14900 / `showable_prices.sales_price` / `discounted_price` / `discount_rate` / `coupon_discounted_price` | 가격·할인·쿠폰 | 🟢 정가·할인·쿠폰 구조화 |
| `is_sold_out` / `is_low_stock` / `stock_threshold`=10 | 재고·신선도 피처 | 🟢 쿠팡은 크롤 HTML만 |
| `category_ids` / `product_origin`="국산" | 분류·원산지 | 🟢 |
| `main_image_url` / `seller_names`=["컬리"] / `is_lowest_price` | ES 카드·최저가 | 동급 |

**엔드포인트(토큰 불필요):** 목록 `GET /v1/categories/{catId}?page_limit=99&page_no={1~10}&delivery_type=0&sort_type=0` (99개/페이지) · 상세 `GET /showroom/v2/products/{no}`.

### 8.3 호출수·속도 제한 (문헌·후기 조사)

- **공식 하드 레이트리밋: 없음** (사설 API, 문서·429 정책 없음). 인증/토큰 불필요.
- **밴 후기: 미보고** — Selenium + `time.sleep(2s~)` 정도로 카테고리 전량 수집, 403/429/IP밴 경험 없음 ([@jo1132](https://velog.io/@jo1132/220909-마켓컬리-크롤링하기), [@ash3767](https://velog.io/@ash3767/마켓컬리-api)).
- **⚠️ 실질 스로틀 = IDC/프록시 IP 차단** — Kurly가 데이터센터·프록시·VPN·Tor 대역을 차단한다는 언급 존재 → **AWS Spot(=IDC IP)에서 직접 폴링 시 차단 위험**. (단 이 검증 박스에서 read API는 200 응답 — 즉시 차단 없었음.)
- 안전 운영: ≤1 req/s·딜레이·소량 바스켓·**비-IDC egress**·honest UA·백오프.

### 8.4 감수·주의 (블로커 아님, 트레이드오프)

- **robots `Disallow: /`(api) 무시** + **ToS** 스크래핑 제한 가능성 → 감수하기로 한 회색 결정.
- **비공식·미문서 사설 API** — `v1`/`v2`/`v3` 버전 churn·`siteName` enum 변동 관측 → 컬리가 바꾸면 수집기 깨짐(취약, 유지보수 부담).
- **제품 적합성:** 컬리는 **프리미엄 새벽배송** 가격대 → "예산 절약 1인가구" 앱의 *기준가*론 편향(높음). 기준 baseline보단 **'프리미엄 옵션 축'**으로.

### 8.5 §6 블로커#2에 대한 함의

쿠팡·지마켓이 🔴인 상황에서 **마켓컬리는 "우회 없이 구조화 가격을 주는" 유일한 실사용 후보**(용량·100g단가·재고까지 정형). 단 위 감수사항 때문에 **"단일 가격축"으론 부적절** — (a) robots/ToS/fragile 리스크, (b) 프리미엄 편향. **권장: 유저 OCR 크라우드소싱(지속가능 기준가) + 마켓컬리 사설API(프리미엄·보강, 저속·비-IDC) 병행.**
